# Phase 2: Add Hono/Mastra TypeScript Backend (sitelens-ai-agent Compatible)

## Background & Motivation

Phase 1 built a Python-based POC (FastAPI + Gradio + SQLite) for local demo and kaizen analysis. Phase 2 adds a **production-ready TypeScript backend** that mirrors the `sitelens-ai-agent` codebase exactly, so the detection feature can be migrated into the team's production Lambda with minimal changes.

This repo is a personal project. The sitelens-ai-agent repo (team production system) is not modified. Instead, the same monorepo structure, tech stack, and code patterns are replicated here. If the product is approved, the `db/`, `app/`, and `infra/` directories can be copied directly into sitelens-ai-agent.

### Why Mirror sitelens-ai-agent

- sitelens-ai-agent is a Mastra-based RAG chatbot deployed as a Lambda function via ECR
- Adding Rekognition to the same Lambda is optimal: Rekognition runs on AWS's side, so the Lambda just sends an API call
- Same codebase, same deployment, no cross-Lambda communication
- Identical patterns = minimal migration friction

**`db/src/models/detection.ts`** - 4 detection tables:

| Table | Purpose | Key Columns |
|-------|---------|-------------|
| `detections` | Image uploads + metadata | userId (FK->users), imageS3Key, rawResponse (jsonb), labelCount |
| `detection_labels` | Per-label results | detectionId (FK->detections, cascade), name, confidence, bbox coordinates |
| `detection_feedback` | Human review | labelId (FK->labels, cascade), isCorrect, expectedLabel, isWrongReason |
| `stress_tests` | Degradation tests | sourceDetectionId, resultDetectionId, labelDiff (jsonb) |

Key design decisions vs Python POC:
- cuid2 text IDs instead of SQLite auto-increment integers (matches sitelens pattern)
- `imageS3Key` instead of `image_path` (images stored in S3, not local filesystem)
- `userId` FK to tie detections to authenticated users
- `jsonb` type for rawResponse and labelDiff (PostgreSQL native JSON with indexing)
- `real` type for confidence scores and bounding box coordinates
- `onDelete: "cascade"` on labels/feedback (parent deletion cascades)
- `onDelete: "set null"` on userId (preserve data if user is deleted)

**`db/src/models/detection-relations.ts`** - Drizzle ORM relations:
- `detections` -> one user, many labels, many feedback, many stress tests (as source and result)
- `detection_labels` -> one detection, many feedback
- `detection_feedback` -> one label, one detection
- `stress_tests` -> one source detection, one result detection (named relations to disambiguate)

**`db/src/models/relations.ts`** - User-side relations:
- `users` -> many detections

**`db/src/index.ts`** - Barrel export for all models, relations, and db connection.

Detection upload flow (`detect.ts`):
1. `c.req.parseBody()` extracts multipart file
2. Validate file type (jpeg/png/webp/gif/bmp) and size (max 15MB)
3. `uploadToS3("detections/{userId}/{timestamp}_{filename}", buffer, contentType)`
4. `detectLabels(buffer, { minConfidence, maxLabels, s3Key })`
5. `saveDetection(...)` to PostgreSQL
6. Return labels with confidence scores and bounding boxes

## Architecture Decisions

### Why a Separate TypeScript Backend (Not Extending the Python POC)

- The production system (sitelens-ai-agent) is TypeScript/Hono/Mastra
- Building the TypeScript version now means migration is a copy operation, not a rewrite
- Python POC remains useful for local demo and Gradio UI (no equivalent in Hono)
- Both coexist in the same repo without interference

### Why No Authentication in the POC

- sitelens-ai-agent uses Cognito authentication via middleware
- This POC skips auth to keep it simple for local testing
- Migration checklist item: wrap routes with `getCognitoUsername(c)` and add userId from auth context

### Why Simplified Logger

- sitelens uses `@mastra/loggers` PinoLogger with request context injection
- This POC uses a simple console-based logger with prefix support
- Same interface (`debug`, `info`, `warn`, `error`) so migration is a drop-in replacement

### S3 Key Strategy for Detection Images

- Pattern: `detections/{userId}/{timestamp}_{filename}`
- Anonymous uploads: `detections/anonymous/{timestamp}_{filename}`
- Stress test results: `stress-tests/{sourceId}/{type}_{level}`
- Timestamp prefix ensures uniqueness without collision risk

### Route Registration Order

Hono matches routes in registration order. Fixed paths must be registered before parameterized paths:
```
/detection/list              (fixed - registered first)
/detection/analysis/summary  (fixed - registered first)
/detection/feedback          (fixed - registered first)
/detection/:detectionId      (parameterized - registered last)
```
Otherwise `/detection/list` would match `:detectionId = "list"`.
